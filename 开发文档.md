# FastRead 开发文档

## 1. 开发环境

- Windows 10/11；
- .NET SDK 8.0 或更新的兼容 SDK；
- PowerShell 5.1+；
- 可选：Visual Studio 2022（安装“.NET 桌面开发”工作负载）。

项目不依赖第三方 NuGet 包。首次构建无需下载业务依赖。

## 2. 目录规划

```text
fastRead/
├─ src/FastRead/                 # Windows Forms 主程序
│  ├─ Core/                      # 领域模型、校验、提示词
│  ├─ Infrastructure/            # Win32、配置、凭据、HTTP
│  └─ UI/                        # 托盘上下文与窗体
├─ tests/FastRead.Tests/         # 无第三方框架的轻量测试程序
├─ 设计文档.md
├─ 开发文档.md
├─ README.md
└─ FastRead.sln
```

## 3. 构建与运行

```powershell
dotnet build FastRead.sln
dotnet run --project src/FastRead/FastRead.csproj
```

运行测试：

```powershell
dotnet run --project tests/FastRead.Tests/FastRead.Tests.csproj
```

发布单文件 x64 程序：

```powershell
dotnet publish src/FastRead/FastRead.csproj -c Release -r win-x64 --self-contained true -p:PublishSingleFile=true
```

## 4. 关键实现约定

### 4.1 线程模型

- WinForms、剪贴板、快捷键窗口必须运行在主 STA 线程；
- 网络请求全程异步，不阻塞 UI 线程；
- SSE 增量内容通过控件的 `BeginInvoke` 回到 UI 线程；
- 每次任务持有独立 `CancellationTokenSource`，新任务会取消旧任务。

### 4.2 全局快捷键

快捷键字符串统一格式为 `Ctrl+Alt+S`。解析器支持 `Ctrl`、`Alt`、`Shift`、`Win` 与一个非修饰键。注册配置时先注销旧热键，若任一新热键失败，则回滚注册旧配置，避免处于半生效状态。

### 4.3 剪贴板

剪贴板可能被其他进程短暂占用，读写需要有限次数重试。恢复是 best-effort：任何恢复失败都不能覆盖任务主错误或导致程序退出。不得把捕获文本写入日志。

### 4.4 HTTP 与流式响应

- 使用单个长期存在的 `HttpClient`；
- 每个请求通过 linked cancellation token 施加用户配置的超时；
- 使用 `ResponseHeadersRead` 尽快开始处理 SSE；
- SSE 按行解析 `data:`，忽略空行与 `[DONE]`；
- 兼容增量字段 `choices[0].delta.content` 与普通字段 `choices[0].message.content`；
- 错误响应最多读取 2,000 字符，且不得包含请求头。

### 4.5 配置迁移

`settings.json` 包含 `schemaVersion`。读取失败或字段缺失时使用默认值并保留可解析字段。未来修改字段含义时递增版本，并在 `SettingsStore` 中显式迁移。

### 4.6 API Key

通过 Windows Credential Manager 的泛型凭据保存：

- Target：`FastRead/ApiKey`；
- Persist：当前用户本地；
- UserName：`FastRead`；
- Blob：UTF-8 编码的 API Key。

设置窗口中的密码框只用于覆盖现有 Key。留空保存表示保留原 Key；“清除密钥”是独立操作。

## 5. 提示词规范

提示词由固定系统指令、语言指令、任务指令、长度约束和原文组成。语言指令来自 `AppSettings.OutputLanguage`：`zh-CN` 生成简体中文，`en` 生成 English。原文使用明确分隔符包裹，并在系统指令中声明：原文中的命令仅是待处理内容，不可改变任务目标，以降低提示注入影响。

总结任务应：

- 保留核心论点、关键事实与结论；
- 删除重复、例证细节和修辞；
- 使用 `AppSettings.OutputLanguage` 指定的语言；默认是简体中文；
- 不补充原文没有的信息。

解释任务应：

- 先给直观含义，再拆解关键概念和逻辑；
- 必要时给简短例子；
- 明确不确定性，不编造背景；
- 使用 `AppSettings.OutputLanguage` 指定的语言；默认是简体中文。

## 6. 测试策略

自动化测试集中在纯逻辑：

- 快捷键解析、格式化与非法组合；
- 配置默认值和校验；
- 提示词任务类型与字符约束；
- Chat Completions 普通响应与 SSE 数据解析；
- 模型错误消息映射。

手工验收覆盖操作系统集成：

1. 在记事本中验证总结和解释；
2. 在浏览器输入框与普通网页文本中验证；
3. 复制一段原始文本，触发取词后确认剪贴板恢复；
4. 同时占用默认快捷键，验证冲突提示；
5. 使用错误 Key、无效 URL、超时服务验证错误提示；
6. 连续快速触发两次，确认第一次被取消且结果不混合；
7. 开启/关闭开机启动并检查当前用户注册表项。

## 7. 代码质量要求

- 启用 nullable 与隐式 using；
- 公共模型使用不可变或受控可变结构；
- UI 层不直接拼接模型请求；
- 不捕获后静默丢弃关键异常；仅剪贴板恢复、托盘释放等清理操作允许 best-effort；
- 所有面向用户的错误均使用中文，内部类型与成员使用英文；
- 提交前执行 Debug/Release 构建与测试程序。

## 8. 发布检查清单

- 更新版本号与 README；
- 运行自动化测试；
- 执行 `dotnet publish`；
- 在未安装 .NET 的干净 Windows 用户环境验证自包含版本；
- 检查发布目录不含 `settings.json`、API Key 或个人数据；
- 手工测试托盘退出、单实例与开机启动。
